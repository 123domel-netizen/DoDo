# Audyt i naprawy — 12 lipca 2026

Pełny przegląd aplikacji + wdrożenie powiadomień push end-to-end.
(Wykonane przez Claude Code na zlecenie; szczegóły techniczne w commitach/diffie.)

## Decyzja architektoniczna: powiadomienia BEZ integracji Google

Cel: powiadomienia na telefonie („za 5 min spotkanie", „deadline projektu"),
**bez** wpisów w Google Calendar i Google Tasks.

Google nie oferuje API „powiadomienie bez wydarzenia w kalendarzu" — jedyną
drogą byłoby tworzenie wydarzeń (czego nie chcemy). Dlatego używamy własnego
**Web Push** (standard W3C): na Androidzie/Chrome jest on i tak dostarczany
przez infrastrukturę Google (FCM), więc „integrację z Google w zakresie
powiadomień" mamy gratis — bez śladu w kalendarzu/zadaniach. Dawna integracja
Google Calendar/Tasks była już wcześniej usunięta z bazy (migracje 0007/0008);
teraz usunięto też jej martwy kod.

## Co zostało WDROŻONE (działa)

1. **Klucze VAPID** — wygenerowane; publiczny w `.env`
   (`VITE_VAPID_PUBLIC_KEY`), oba jako sekrety funkcji
   (`supabase secrets set VAPID_PUBLIC_KEY/VAPID_PRIVATE_KEY/VAPID_SUBJECT`).
2. **Edge Function `send-reminders`** — przebudowana i wdrożona (deploy OK,
   smoke test `{"ok":true,"sent":0}`):
   - filtruje tombstony (`deleted_at`) — wcześniej przypominała o usuniętych,
   - **rozwija cykle (RRULE)** — przypomnienie dla każdego wystąpienia, nie
     tylko pierwszego; dedupe per wystąpienie przez `reminder_log(fire_at)`,
   - **deadline** (`payload.deadlineAt`): push 24 h przed i w chwili terminu,
   - **przypomnienia osobiste uczestników SHARE**
     (`item_participants.personal_reminders`) — wysyłane do uczestnika,
   - czasy w treści formatowane w strefie **Europe/Warsaw** (wcześniej UTC),
   - okno zapytania poszerzone do 45 dni (długie offsety przypomnień),
   - uwaga techniczna: `rrule` importowane z `esm.sh` — import `npm:rrule`
     powodował BOOT_ERROR na Edge Runtime.
3. **pg_cron + pg_net włączone, job `send-reminders-every-minute`** (co minutę)
   utworzony przez Management API. Funkcja ma `verify_jwt=false`, więc cron
   woła ją bez service role key w SQL (bez sekretu w bazie).
4. **Klient — `enablePush()`** naprawione: `upsert(..., { onConflict: "endpoint" })`
   (wcześniej ponowna rejestracja urządzenia wysypywała się na unique i w bazie
   zostawały martwe klucze), obsługa błędów, wymóg zalogowania.
5. **Dzwonek powiadomień** (desktop + mobile) — zawsze pokazuje wynik:
   „push włączony" / „tylko lokalne, bo …" / błąd. Wcześniej cichy fallback
   udawał, że push działa.
6. **Lokalny scheduler przypomnień** — przepisany (`src/lib/reminderScheduler.ts`
   + `useReminderScheduler`):
   - cykle per wystąpienie (dedupe w localStorage, TTL 7 dni),
   - deadline'y lokalnie (24 h przed + w terminie),
   - okno doręczenia spóźnionych do 60 min (np. po wybudzeniu laptopa),
   - gdy urządzenie ma aktywną subskrypcję push → lokalne odpalanie pomijane
     (serwer dosyła; koniec podwójnych powiadomień).
7. **Service worker** — nawigacje network-first (po deployu od razu nowa
   wersja), zasoby cache-first jak dotąd.
8. **Martwy kod Google usunięty**: funkcje `google-oauth`/`google-sync`/
   `google-webhook` + `_shared` (ok. 2200 linii) — skasowane lokalnie
   **i w chmurze** (functions delete), sekrety `GOOGLE_*` usunięte,
   `config.toml`/README/`.env.example`/`vite-env.d.ts` wyczyszczone.
   Pola `google*` w `types.ts` zostają — silnik cykli używa `googleRecurrence`
   jako generycznego nośnika RRULE.
9. **Testy: vitest, 38 testów** (`npm test`) — merge/sync (`mergeItemOnSync`,
   tombstony), rozwijanie cykli, reguły RRULE, scheduler przypomnień
   (cykle, deadline, SHARE, okna czasowe, fired-log).
10. **Porządki repo**: `supabase/.temp/` i `dev-dist/` odtrackowane + w
    `.gitignore`.

## Co musisz zrobić SAM (krótko)

1. **Na telefonie (Android):** otwórz aplikację w Chrome → zainstaluj PWA
   (menu → „Zainstaluj aplikację") → w aplikacji kliknij **dzwonek** →
   zgódź się na powiadomienia. Powinien pojawić się komunikat
   „Powiadomienia push włączone…".
   **iPhone:** Safari → Udostępnij → „Dodaj do ekranu początkowego" → otwórz
   z ekranu początkowego → dzwonek (wymaga iOS 16.4+).
2. **Test:** dodaj wydarzenie za ~10 min z przypomnieniem „5 min przed",
   zamknij aplikację — push powinien przyjść na telefon.
3. Po wejściu na komputerze też kliknij dzwonek (każde urządzenie rejestruje
   się osobno).

## Znane ograniczenia / następne kroki (propozycje)

- **Załączniki plikowe** są zapisywane jako base64 w JSONB (`payload`) —
  do 5 MB na plik; sync ciągnie całość przy każdej zmianie. Docelowo:
  Supabase Storage + podpisane linki. (Największa pozostała rzecz.)
- **Monolity do rozbicia:** `ItemEditorPanel.tsx` (~1700 linii),
  `cloud.ts` (~980). Sensowne przy następnej większej zmianie edytora.
- **ESLint + CI** (GitHub Actions: vitest + tsc + build) — repo nie ma
  remote'a; dodać przy okazji założenia repo na GitHubie.
- Dedupe lokalny vs push opiera się na obecności subskrypcji push na
  urządzeniu — jeśli kiedyś wyłączysz cron, urządzenia z subskrypcją nie
  dostaną też lokalnych (do ewentualnej zmiany strategii).
- `reminder_log` rośnie (1 wpis / wysłane powiadomienie) — przy okazji dodać
  czyszczenie wpisów starszych niż ~30 dni (mały cron SQL).
- Widok „11 dni" a README „9 dni" — dokumentacja częściowo doszlifowana,
  warto przejrzeć całość README po zmianach produktowych.

## Jak to jest spięte (ściąga operacyjna)

- Frontend: `VITE_VAPID_PUBLIC_KEY` w `.env` (build wymaga rebuildu po zmianie).
- Sekrety funkcji: `supabase secrets list` (VAPID_*).
- Deploy funkcji: `supabase functions deploy send-reminders --no-verify-jwt`.
- Cron: job `send-reminders-every-minute` w `cron.job`; diagnostyka:
  `select * from cron.job_run_details order by start_time desc limit 5;`
  oraz `select status_code from net._http_response order by id desc limit 5;`
- Log wysyłek/dedupe: tabela `reminder_log`.
- Subskrypcje urządzeń: tabela `push_subscriptions` (wygasłe 404/410 są
  usuwane automatycznie przy wysyłce).
